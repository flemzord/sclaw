Every sclaw agent starts with built-in tools registered in the global tool registry. These provide the baseline capabilities agents need to interact with the filesystem, execute commands, and manage configuration.

## exec

Execute a shell command in the agent's workspace.

| Property | Value |
|----------|-------|
| **Scope** | `exec` |
| **Default policy** | `allow` |

### Schema

```json
{
  "command": "string (required)",
  "timeout_seconds": "integer (optional, default 30, max 600)"
}
```

### Behavior

- Runs `sh -c <command>` with the working directory set to the agent's workspace
- Uses `SanitizedEnv` — sensitive environment variables are stripped before execution
- Captures stdout and stderr (capped at 1 MiB each)
- Non-zero exit codes return `IsError: true` with stderr content so the agent can react
- Commands that exceed the timeout are killed

<Warning>
The `exec` tool does **not** run inside a Docker sandbox by default. Enable sandboxing in the security configuration for production deployments where agents execute untrusted commands.
</Warning>

### Examples

```json
{"command": "gh issue list --limit 5"}
```

```json
{"command": "remindctl add 'Review PR #42'", "timeout_seconds": 10}
```

## read_file

Read the contents of a file in the workspace, data directory, or allowed directories.

| Property | Value |
|----------|-------|
| **Scope** | `read_only` |
| **Default policy** | `allow` |

### Schema

```json
{
  "path": "string (required)"
}
```

### Behavior

- Paths can be relative (resolved against workspace) or absolute
- Resolution order: workspace → data directory → allowed directories (RO or RW)
- Symlinks are resolved — a symlink pointing outside all allowed boundaries is rejected
- Directories are rejected (use `exec` with `ls` instead)
- Files larger than 1 MiB are rejected to avoid blowing up the LLM context window

### Path Safety

All path operations go through `SafePathForRead`, which tries each location in order:

1. **Workspace** — `SafePath` cleans the path, resolves symlinks, verifies containment
2. **Data directory** — same checks, for the agent's persistent data directory
3. **Allowed directories** — resolves the absolute path, validates via `security.ValidatePath`, then checks `PathFilter.CheckRead` (RO or RW)

Each step includes:
- Path cleaning (`filepath.Clean`)
- Symlink resolution (`filepath.EvalSymlinks`)
- Separator-aware prefix comparison to prevent `/data` matching `/data-staging`
- `security.ValidatePath` to block `/proc`, `/sys`, `/dev`

<Note>
The `PathFilter` resolves symlinks on configured directory paths at creation time. This ensures consistent matching on systems where paths like `/var` are symlinks to `/private/var` (macOS).
</Note>

## write_file

Write content to a file in the workspace or read-write allowed directories.

| Property | Value |
|----------|-------|
| **Scope** | `read_write` |
| **Default policy** | `allow` |

### Schema

```json
{
  "path": "string (required)",
  "content": "string (required)"
}
```

### Behavior

- Resolution order: workspace → allowed directories (RW only)
- Creates parent directories automatically (`os.MkdirAll` with `0755`)
- Writes files with `0644` permissions
- Content larger than 1 MiB is rejected
- Read-only allowed directories are never writable — only `rw` directories accept writes

### Examples

```json
{"path": "notes/standup.md", "content": "# Standup 2025-01-15\n- Reviewed PR #42\n- Fixed CI flake"}
```

## config_get

Read the current configuration and compute a hash for concurrency control.

| Property | Value |
|----------|-------|
| **Scope** | `read_only` |
| **Default policy** | `allow` |

### Schema

No parameters required.

### Behavior

- Reads the config file from disk
- Computes a SHA-256 hash on the raw bytes (before env expansion) for deterministic results
- Redacts all secrets before returning
- Returns the config as JSON alongside the `base_hash`

### Output

```json
{
  "config": { "version": "1", "modules": { ... } },
  "base_hash": "sha256_hex_digest"
}
```

The `base_hash` is required by `config_patch` and `config_apply` for optimistic concurrency control.

## config_validate

Dry-run validation of YAML configuration without writing to disk.

| Property | Value |
|----------|-------|
| **Scope** | `read_only` |
| **Default policy** | `allow` |

### Schema

```json
{
  "yaml_content": "string (required)"
}
```

### Behavior

- Expands environment variables and parses YAML
- Runs full schema validation (version, modules, agents)
- Returns `valid: true` or `valid: false` with the error message
- No side effects — purely informational

## config_patch

Merge a partial YAML patch into the current configuration.

| Property | Value |
|----------|-------|
| **Scope** | `read_write` |
| **Default policy** | `ask` |

### Schema

```json
{
  "base_hash": "string (required, from config_get)",
  "patch": "string (required, partial YAML to merge)"
}
```

### Behavior

- Verifies the `base_hash` matches the current file — rejects if the config changed since the last read
- Merges using RFC 7386 JSON Merge Patch semantics on the YAML AST:
  - Mapping keys are merged recursively (patch overrides)
  - Sequences are replaced entirely
  - `null` values delete keys
- Validates the merged result before writing
- Performs an atomic write (temp file + rename)
- Triggers a hot-reload after success
- Warns if the plugin list changed (rebuild may be required)
- Preserves `${VAR}` references, comments, and key ordering

## config_apply

Replace the entire configuration with new YAML content.

| Property | Value |
|----------|-------|
| **Scope** | `read_write` |
| **Default policy** | `ask` |

### Schema

```json
{
  "base_hash": "string (required, from config_get)",
  "yaml_content": "string (required, complete YAML)"
}
```

### Behavior

- Same concurrency control as `config_patch` via `base_hash`
- Full replacement instead of merge
- Validates the new content before writing
- Atomic write + hot-reload

### Typical Workflow

```
1. config_get        → read config + base_hash
2. config_validate   → dry-run check (optional)
3. config_patch      → merge partial changes
   or config_apply   → replace entirely
4. (hot-reload triggers automatically)
```

## Security Guarantees

All built-in tools enforce consistent security boundaries:

| Protection | Mechanism |
|-----------|-----------|
| **Path traversal** | `SafePath` rejects `../` escape, symlinks pointing outside workspace, and absolute paths outside allowed boundaries |
| **Directory access control** | `PathFilter` enforces per-directory RO/RW permissions for `read_file` and `write_file` |
| **System paths** | `security.ValidatePath` blocks `/proc`, `/sys`, `/dev` |
| **Secret leakage** | `exec` uses `SanitizedEnv` — API keys, tokens, and credentials are stripped |
| **Resource limits** | 1 MiB cap on file reads, file writes, command output, and config content |
| **Timeout** | `exec` defaults to 30s, capped at 10 minutes |
| **Concurrency control** | `config_patch`/`config_apply` use SHA-256 hash to prevent lost updates |
| **Secret redaction** | `config_get` redacts all secrets before returning config to the LLM |
| **Atomic writes** | Config changes use temp file + rename to prevent partial writes |

## Approval Policies

By default, all built-in tools use `allow` — they execute without prompting the user. Override this per-context in the configuration:

```yaml
security:
  policy:
    dm:
      tools:
        exec: ask        # Require approval for shell commands in DMs
        write_file: ask  # Require approval for file writes in DMs
    group:
      default: deny      # Block all tools in group chats by default
```

See [Tools](/concepts/tools) for the full approval system documentation.

## Module Migration

The three core tools (`exec`, `read_file`, `write_file`) are now available as configurable modules in addition to their built-in versions. This allows you to customize timeouts, size limits, and approval policies without modifying code.

| Built-in Tool | Module ID | Module Package |
|---------------|-----------|----------------|
| `exec` | `tool.shell` | `modules/tool/shell/` |
| `read_file` | `tool.file_read` | `modules/tool/file_read/` |
| `write_file` | `tool.file_write` | `modules/tool/file_write/` |

### How It Works

- **No `tool.*` modules configured** — the legacy built-in tools are used automatically. Fully backward compatible, no changes needed.
- **Some `tool.*` modules configured** — each configured module replaces its corresponding built-in. Unconfigured tools still use builtins. For example, if only `tool.shell` is in your config, `exec` comes from the module while `read_file` and `write_file` remain built-in.
- **All `tool.*` modules configured** — no built-in tools are loaded; everything is module-driven.

```yaml
# Example: configure all three tool modules
modules:
  tool.shell:
    timeout: "1m"
    default_policy: "ask"
  tool.file_read:
    max_file_size: 5242880
  tool.file_write:
    create_dirs: true
```

See [Module Configuration](/configuration/modules#toolshell) for the full configuration reference.

## Registration

Built-in tools are registered automatically during application startup. The wiring logic first discovers tools from `ToolProvider` modules, then fills in any gaps with built-in tools. Config tools are registered via `configtool.RegisterAll()` with their dependencies (config path, redactor, reload function). All tools are available to every agent through the global tool registry.

```go
// internal/tool/builtin/register.go
func RegisterAll(registry *tool.Registry) error
func All() []tool.Tool  // Returns instances without registering

// internal/tool/configtool/configtool.go
func RegisterAll(registry *tool.Registry, deps Deps) error
```

To add a custom built-in tool, implement the `tool.Tool` interface and register it through the appropriate package.

## LLM Visibility

Built-in tools are automatically advertised to the LLM on every request. During **Step 9 (Context Assembly)** of the message pipeline, the router calls `loop.ToolDefinitions()` which collects all registered tools — including built-in ones — and injects them into the provider request as an OpenAI-compatible `tools` array.

This means the LLM knows about `exec`, `read_file`, and `write_file` and can call them during the ReAct reasoning loop. No additional configuration is needed — registering a tool is sufficient for it to appear in the LLM's tool list.

The pipeline also injects directory context into the system prompt: the **workspace directory path** (Step 9c) and any **allowed directories** with their access modes (Step 9d). This lets the LLM know exactly which directories it can access and whether they are read-only or read-write.

See [Tools](/concepts/tools#tool-advertisement-to-the-llm) for details on how tool definitions flow through the system.
