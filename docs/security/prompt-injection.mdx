Indirect prompt injection is one of the most critical risks for AI agent systems. An attacker can embed malicious instructions inside user messages, tool outputs, or external data sources to manipulate the agent into taking unintended actions.

## Threat Model

### Attack Vectors

| Vector | Description |
|--------|-------------|
| **User message injection** | Direct malicious instructions in chat messages. |
| **Tool output injection** | Crafted responses from external APIs/tools. |
| **Context poisoning** | Malicious content injected via conversation history. |
| **Data exfiltration** | Instructions to leak credentials via tool calls. |

## Mitigations

| Risk | Mitigation | Component |
|------|-----------|-----------|
| Credential leakage via prompts | `CredentialStore` with context injection — secrets never in LLM context | `security/credentials.go` |
| Secret leakage in logs | `RedactingHandler` wraps all slog output | `security/sloghandler.go` |
| Secret leakage via subprocess env | `SanitizedEnv()` strips sensitive vars | `security/env.go` |
| API key patterns in outputs | `Redactor` with compiled regex patterns | `security/redactor.go` |
| Tool abuse / rapid exploitation | `RateLimiter` with sliding windows | `security/ratelimit.go` |
| JSON bombs / large payloads | `ValidateMessageSize` + `ValidateJSONDepth` | `security/validation.go` |
| Unauthorized tool execution | Policy-based approval (allow/ask/deny) | `tool/tool.go` |
| Cross-session data access | Session isolation + ParentID validation | `subagent/manager.go` |
| Dangerous filesystem access | `ValidatePath` blocks `/proc` access | `security/env.go` |
| Unrestricted network access | `URLFilter` with default-deny | `security/urlfilter.go` |
| Unaudited security events | `AuditLogger` covers all event types | `security/audit.go` |
| Webhook body bombs | `http.MaxBytesReader` on dispatcher | `gateway/webhook.go` |

## Design Principles

### 1. Never Pass Secrets to LLM Context

Secrets are managed exclusively through `CredentialStore` and accessed via `context.Context`. They are never embedded in system prompts, user messages, or tool outputs that flow through the LLM.

### 2. Defense in Depth

Multiple layers protect against credential leakage:

<Steps>

### Layer 1: CredentialStore

Secrets are stored in a dedicated store and never reach the LLM. Modules access them via context injection.

### Layer 2: Redactor

Pattern-based detection catches known credential formats (OpenAI, Anthropic, GitHub, AWS, Slack keys) in any string.

### Layer 3: RedactingHandler

The log handler intercepts all `slog` output and redacts detected patterns before writing.

### Layer 4: SanitizedEnv

Subprocess environments are stripped of sensitive variables before execution.

### Layer 5: AuditLogger

Full audit trail with redaction provides post-incident investigation capability.

</Steps>

### 3. Fail-Closed

- Empty URL allow-lists block all domains (default-deny)
- Sandbox executor returns error if Docker unavailable (fail-closed)
- Rate limiter rejects requests when limits are exceeded
- Message validation rejects oversized or deeply nested payloads

### 4. Minimal Privilege

- Tools declare required scopes at registration
- Policy system controls execution based on context (DM vs group)
- Subprocess environments stripped of sensitive variables
- Sandbox containers run read-only with network disabled

## Remaining Risks

<Accordion title="Novel credential formats">
While sclaw redacts known patterns, novel credential formats may bypass regex detection. **Mitigation:** Keep patterns updated, monitor audit logs for anomalies.
</Accordion>

<Accordion title="Timing side-channels">
Rate limiting uses wall-clock time which could leak information about internal processing. **Mitigation:** Constant-time comparison for auth (already implemented).
</Accordion>

<Accordion title="Resource exhaustion">
Sophisticated attacks could consume significant compute within rate limits. **Mitigation:** Monitor metrics, adjust limits based on usage patterns.
</Accordion>

<Accordion title="Model-level jailbreaks">
These are outside sclaw's control and depend on the underlying LLM provider. **Mitigation:** Defense in depth via tool approval system — even if the model is compromised, dangerous tools require explicit user approval.
</Accordion>
