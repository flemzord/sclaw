sclaw's credential management system ensures that secrets (API keys, tokens, passwords) never reach the LLM context, never appear in logs, and never leak to subprocesses.

## CredentialStore

The `CredentialStore` is the central secret management component. It stores credentials in memory and makes them available to modules via `context.Context` — never via the LLM prompt.

### Interface

| Method | Description |
|--------|-------------|
| `Set(name, value)` | Store a credential. |
| `Get(name)` | Retrieve a credential by name. |
| `Has(name)` | Check if a credential exists. |
| `Names()` | List all stored credential names. |
| `Values()` | Return all credential values (for redaction patterns). |
| `Delete(name)` | Remove a credential. |
| `Len()` | Return the number of stored credentials. |

### Context Injection

Credentials are injected into the request context and extracted by modules that need them:

```go
// Inject credentials into context
ctx = security.WithCredentials(ctx, store)

// Extract credentials in a module
creds := security.CredentialsFromContext(ctx)
apiKey := creds.Get("openai_api_key")
```

<Warning>
Never embed credentials in system prompts, tool descriptions, or any content that passes through the LLM. Always use the CredentialStore via context injection.
</Warning>

## Redaction

### Pattern Detection

The `Redactor` uses compiled regular expressions to detect known credential formats:

| Provider | Pattern |
|----------|---------|
| OpenAI | `sk-[a-zA-Z0-9]{20,}` |
| Anthropic | `sk-ant-[a-zA-Z0-9]{20,}` |
| GitHub | `ghp_[a-zA-Z0-9]{36}`, `gho_*`, `ghu_*`, `ghs_*`, `ghr_*` |
| AWS | `AKIA[0-9A-Z]{16}` |
| Slack | `xoxb-*`, `xoxp-*`, `xoxo-*`, `xoxa-*` |

Detected credentials are replaced with `[REDACTED]`.

### Log Redaction

The `RedactingHandler` wraps Go's `slog.Handler` to intercept all log output:

- Every string attribute is passed through the `Redactor`
- Known credential patterns are replaced before the log is written
- This catches accidental leaks in error messages, debug output, etc.

<Note>
Never use `fmt.Println` or raw `slog` for secret-bearing data. Always log through the `RedactingHandler`-wrapped logger.
</Note>

## Environment Sanitization

### SanitizedEnv

When sclaw spawns subprocesses (for tool execution), it uses `SanitizedEnv()` instead of `os.Environ()`:

```go
env := security.SanitizedEnv()
cmd.Env = env
```

This strips known sensitive environment variables:
- `*_API_KEY`
- `*_TOKEN`
- `*_SECRET`
- `*_PASSWORD`
- AWS, GCP, and Azure credentials

### Path Validation

`ValidatePath` blocks access to sensitive filesystem paths:

- `/proc` — Linux process information
- `/sys` — Linux system information
- Other sensitive system paths

## Best Practices

<Accordion title="For Configuration Authors">
- Always use `${VAR}` syntax for secrets in YAML — never hardcode values
- Use `api_key_env` when possible to reference variables by name
- Set `allow_users` on channels to restrict access
- Enable rate limits in production
</Accordion>

<Accordion title="For Module Developers">
- Access secrets via `CredentialStore`, never from environment or config
- Use `SanitizedEnv()` for subprocesses, never `os.Environ()`
- Log through the provided logger (already wrapped with `RedactingHandler`)
- Validate all external input at system boundaries
</Accordion>
