The `agents` section enables multi-agent mode. Each key is an agent name, and its value configures routing, provider binding, and loop parameters. When `agents` is omitted, sclaw runs in **single-agent mode** with a default agent.

## Agent Fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `data_dir` | string | `{DataDir}/agents/{name}` | Persistent data directory for this agent. |
| `workspace` | string | — | Working directory for tool execution. |
| `allowed_dirs` | list | — | Additional directories accessible outside the workspace. |
| `provider` | string | — | Module ID of the provider to use (must exist in `modules`). |
| `tools` | list | — | Tool names available to this agent. |
| `exclude_skills` | list | — | Global skill names to exclude for this agent. |
| `streaming` | bool | `true` | Stream LLM responses progressively to the channel. |
| `memory` | object | — | Memory settings for this agent. |
| `routing` | object | — | Routing rules for message dispatch. |
| `loop` | object | — | ReAct loop parameter overrides. |

## Routing

Routing rules determine which agent handles an incoming message. The router evaluates agents **in declaration order** and picks the first match. At most one agent may be marked as `default`.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `channels` | list | — | Channel names this agent handles (e.g., `["telegram"]`). |
| `users` | list | — | User IDs this agent responds to. |
| `groups` | list | — | Group IDs this agent responds to. |
| `default` | bool | `false` | Fallback agent when no other routing rule matches. |

<Warning>
Only one agent may set `default: true`. If no agent matches and there is no default, the message is dropped.
</Warning>

## Loop Parameters

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `max_iterations` | int | `0` | Maximum ReAct loop iterations (0 = unlimited). |
| `token_budget` | int | `0` | Token budget for the agent loop (0 = unlimited). |
| `timeout` | string | — | Maximum duration for a single agent invocation (e.g., `"5m"`). |
| `loop_threshold` | int | `0` | Consecutive identical actions before breaking the loop. |

<Tip>
Set `loop_threshold` to 3–5 to prevent the agent from getting stuck in repetitive tool call patterns. The loop detector tracks consecutive identical action signatures.
</Tip>

## Memory

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | bool | `true` | Whether conversation history is persisted for this agent. |

When `enabled` is `false`, no SQLite database is opened for the agent, and conversation history is only kept in memory for the duration of the session.

## Streaming

Streaming is **enabled by default**. When active, the agent loop streams LLM responses progressively to the channel instead of waiting for the complete response. Channels that support streaming (e.g., Telegram) edit the message in place as chunks arrive, giving users real-time feedback.

If the channel does not support streaming, sclaw automatically falls back to the standard synchronous send — no configuration change needed.

To disable streaming for a specific agent:

```yaml
agents:
  main:
    streaming: false
```

## Allowed Directories

By default, `read_file` and `write_file` can only access files inside the agent's `workspace`. The `allowed_dirs` field grants access to additional directories outside the workspace with granular permissions.

Each entry specifies a `path` and a `mode`:

| Mode | Description |
|------|-------------|
| `ro` | Read-only access (default). The agent can read files but not write. |
| `rw` | Read-write access. The agent can both read and write files. |

```yaml
agents:
  coder:
    workspace: "/tmp/agent-scratch"
    allowed_dirs:
      - path: "/home/user/projects/monrepo"
        mode: "ro"
      - path: "/home/user/documents"
        mode: "rw"
```

In this configuration, the agent can:
- Read and write freely in `/tmp/agent-scratch` (workspace)
- Read files in `/home/user/projects/monrepo` (read-only)
- Read and write files in `/home/user/documents` (read-write)

<Warning>
Without `allowed_dirs`, giving an agent access to files outside its workspace requires setting `workspace: "/"` — which is a security risk. Use `allowed_dirs` instead for fine-grained access control.
</Warning>

<Note>
The `exec` tool is not affected by `allowed_dirs` — it always runs with `cwd` set to the workspace. Shell commands can naturally access any path the OS allows. `allowed_dirs` only controls `read_file` and `write_file`.
</Note>

### Directory Access Model

sclaw uses three directory levels per agent:

| Directory | Access | Purpose |
|-----------|--------|---------|
| `workspace` | Full read-write (tools + exec cwd) | Sandbox for the agent's work |
| `allowed_dirs` | RO or RW per entry (read_file / write_file only) | Controlled access to external directories |
| `data_dir` | Internal only (not exposed to tools) | SOUL.md, skills/, memory storage |

### Path Resolution Order

When `read_file` resolves a path, it tries locations in this order:

1. **Workspace** — relative and absolute paths within the workspace
2. **Data directory** — absolute paths within the data directory
3. **Allowed directories** — absolute paths within any allowed directory (RO or RW)

When `write_file` resolves a path:

1. **Workspace** — relative and absolute paths within the workspace
2. **Allowed directories** — absolute paths within RW directories only

## Single-Agent Mode

When the `agents` section is omitted entirely, sclaw runs in single-agent mode:

```yaml
version: "1"

modules:
  memory.sqlite: {}
  channel.telegram:
    token: "${TELEGRAM_BOT_TOKEN}"
  provider.openai_compatible:
    base_url: "https://api.openai.com/v1"
    api_key: "${OPENAI_API_KEY}"
    model: "gpt-4o"
    context_window: 128000
```

## Multi-Agent Example

```yaml
agents:
  main:
    provider: "provider.openai_compatible"
    memory:
      enabled: true
    routing:
      channels: ["telegram"]
      default: true
    loop:
      max_iterations: 10
      timeout: "5m"

  coder:
    provider: "provider.openai_compatible"
    workspace: "/tmp/coder-scratch"
    allowed_dirs:
      - path: "/home/user/projects"
        mode: "ro"
      - path: "/home/user/output"
        mode: "rw"
    tools: ["exec", "read_file", "write_file"]
    memory:
      enabled: true
    routing:
      users: ["123456789"]
    loop:
      max_iterations: 25
      timeout: "10m"
```

In this setup:
- **main** handles all Telegram messages by default
- **coder** takes over when user `123456789` sends a message, with a sandboxed workspace, read-only access to the source code, and a separate output directory

<Note>
Each agent gets its own data directory, SOUL.md personality file, skills directory, and (optionally) its own SQLite memory database. See [Agent Personality](/concepts/soul), [Skills](/concepts/skills), and [Memory System](/concepts/memory) for details.
</Note>
