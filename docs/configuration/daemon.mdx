sclaw can run as a system service (daemon) managed by your operating system's service manager. This uses [kardianos/service](https://github.com/kardianos/service) to provide cross-platform support.

## Supported Platforms

| Platform | Service Manager |
|----------|----------------|
| Linux | systemd, Upstart, SysV |
| macOS | launchd |
| Windows | Windows Service |

## User vs System Service

By default, sclaw installs as a **user-level service** (LaunchAgent on macOS, user systemd on Linux). This does not require root and runs under your user account.

To install as a **system-level service** (LaunchDaemon / system systemd), add `--system`:

```bash
sudo sclaw service install --system --config /etc/sclaw/sclaw.yaml
```

<Warning>
The `--system` flag must be consistent across all `service` subcommands. If you installed with `--system`, you must also use `--system` for start, stop, status, and uninstall.
</Warning>

## Commands

### Install

Register sclaw as a service:

```bash
# User-level (default, no root needed)
sclaw service install --config ~/.config/sclaw/sclaw.yaml

# System-level (requires root)
sudo sclaw service install --system --config /etc/sclaw/sclaw.yaml
```

### Start

Start the service via the service manager:

```bash
sclaw service start
```

### Stop

Stop the running service:

```bash
sclaw service stop
```

### Status

Check whether the service is running:

```bash
sclaw service status
```

### Uninstall

Remove the service registration:

```bash
sclaw service uninstall
```

## How It Works

When installed, the service manager is configured to run `sclaw start` with the provided flags. The service wrapper handles:

- **Startup**: launches the full sclaw application in the background
- **Shutdown**: gracefully stops all modules when the service manager requests it
- **Hot reload**: `SIGHUP` and file-watcher reload continue to work as in interactive mode

<Note>
The `--config` flag passed to `sclaw service install` is baked into the service definition. If you move your config file, you need to uninstall and reinstall the service.
</Note>

## Environment Variables

When running as a service, the process does **not** inherit your shell environment — launchd and systemd use a minimal default `PATH` that excludes locations like `/opt/homebrew/bin` or `/usr/local/bin`.

sclaw handles this automatically:

- **PATH** is always captured from your shell at `service install` time and baked into the service definition.
- **Custom variables** can be declared in the `service.env` section of your config file.

### `service.env`

Use this section to inject environment variables into the service definition:

```yaml
version: "1"

service:
  env:
    GOG_ACCOUNT: "you@gmail.com"
    LANG: "en_US.UTF-8"
    EDITOR: "vim"
    # 1Password references work here too
    SOME_TOKEN: "op://MyVault/sclaw/token"
    # Environment variable expansion works
    HOME_OVERRIDE: "${HOME}"
```

These values are resolved at config load time (environment variable expansion and `op://` resolution happen before injection) and written into the service definition (launchd plist / systemd unit).

<Note>
Changes to `service.env` require a reinstall of the service to take effect:

```bash
sclaw service uninstall
sclaw service install --config ~/.config/sclaw/sclaw.yaml
sclaw service start
```
</Note>

### PATH Handling

| Scenario | Behavior |
|----------|----------|
| No `service.env` | PATH is auto-captured from your shell |
| `service.env` without PATH | PATH is auto-captured, other vars are added |
| `service.env` with explicit PATH | Your explicit PATH is used (no auto-capture) |

<Warning>
If you override PATH explicitly, make sure it includes the directories for all CLI tools your agents use (e.g. `gog`, `op`, `gh`).
</Warning>

### What `service.env` Does NOT Do

The `service.env` section only affects the **service definition** (launchd plist or systemd unit). It does not:

- Set variables for interactive mode (`sclaw start`) — those come from your shell
- Affect hot reload — changing `service.env` and sending SIGHUP has no effect on the running process environment
- Replace `${VAR}` expansion in other config sections — use standard YAML `${VAR}` syntax for that

## Logs

In daemon mode, logs go to the system log facility:

- **Linux (systemd)**: `journalctl -u sclaw`
- **macOS (launchd)**: `log show --predicate 'process == "sclaw"'`
- **Windows**: Event Viewer

<Info>
When running interactively (`sclaw start`), logs still go to stderr as usual. The system logger is only used when running as a managed service.
</Info>
